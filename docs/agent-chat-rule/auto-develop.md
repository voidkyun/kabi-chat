# Agent自律開発ワークフロー

## 有効条件

このルールは、ユーザーの発言に下記キーワードが含まれる場合のみ有効とする。

`[autodev]`

詳細な実行手順は `docs/dev-workflow/autodev.md` を参照する。  
詳細なレビュー基準は `docs/dev-workflow/code-review.md` を参照する。

## 基本方針

`[autodev]` が有効な場合、Agentはユーザー提案を起点に、必要なGitHub操作、実装、レビュー、デプロイ後報告までを自律的に行う。

ただし、以下に該当する変更はユーザー承認を得るまで実行してはならない。

- インフラコストの増減を伴う変更
- GitHub上の設定変更
- AWS, Discord Developer Portal など外部サービス上の設定変更
- 認証/認可に関わる変更
- DB schema migration または不可逆なデータ変更
- API互換性を壊す変更
- Secret, CI/CD, 権限設定に関わる変更
- 影響範囲が広く、仕様変更を含む可能性が高い変更

## 実行フロー

Agentは原則として以下の順で作業する。

1. ユーザー提案を整理し、実装プランを作成する
2. GitHub上の既存Issue / Pull Request / 関連コードを確認する
3. 必要なIssueが不足している場合は追加・整理する
4. 実装を行う
5. 必要なテストを追加または更新する
6. lint / typecheck / test / build のうち関係する検証を実行する
7. Pull Requestを作成する
8. 自己レビューを行う
9. 指摘があれば修正し、再検証する
10. merge条件を満たす場合のみ main にマージする
11. CI/CD によるデプロイ後、可能な範囲で smoke check を行う
12. ユーザーにパッチノート形式で報告する

初回PR、review指摘への対応PR、follow-up PR を区別せず、main にマージする前のすべてのPRで上記のレビュー工程を省略してはならない。ユーザーから「CI確認して続けて」「そのまま進めて」などの指示があっても、自己レビュー前に merge へ進んではならない。

merge 前には、Agent自身が「自己レビューを完了したこと」「重大指摘の有無」「残っている merge blocker の有無」をユーザーに明示しなければならない。この明示が終わる前に merge してはならない。

## merge条件

以下をすべて満たす場合のみ、Agentは main へのマージを行ってよい。

- 関係する検証がすべて成功している
- 対象PRに紐づく必要な check run が成功している
- 自己レビューで重大な指摘が残っていない
- 承認必須条件に該当しない
- 変更が小〜中規模であり、影響範囲を説明可能である
- デプロイ後の失敗時に切り戻し可能である

legacy status API の `pending` や空応答だけを根拠に merge 判定をしてはならない。GitHub上で確認できる check run を優先して判定する。

上記を満たさない場合、AgentはPR作成までで停止し、ユーザーに状況を報告する。

## 自己レビュー

自己レビューでは、1回のレビューごとにSubAgentを明示的にSpawnする。  
レビュー用Agentには、余計な実装背景や意思決定の正当化を極力渡さず、現在の差分と必要最小限の周辺コードのみをもとに第三者視点でレビューさせる。

少なくとも以下の観点ごとにレビューを行う。

- correctness
- security
- test gaps
- maintainability

必要であれば、performance や race condition も追加する。

レビュー結果は親Agentが集約し、修正の要否を判断する。

## ユーザーへの報告形式

最終報告はパッチノート形式とし、少なくとも以下を含める。

- 変更内容
- 変更理由
- 影響範囲
- 実施した検証
- 残課題 / 今後の改善候補
