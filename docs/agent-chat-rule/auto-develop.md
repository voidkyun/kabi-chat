# `autodev` 発話の解釈ルール

## 位置づけ

この文書は `autodev` を起動するユーザー発話をどう解釈するかを定義するルールである。

- 発火条件
- 実行してよい責務の範囲
- 承認が必要な変更
- merge 前に必ず守る制約

具体的な実行手順は skill として `docs/dev-workflow/autodev.md` に分離する。  
レビュー手順の詳細は `docs/dev-workflow/code-review.md` を参照する。

skill とルールが衝突する場合は、このルールを優先する。

## 発火条件

このルールは、ユーザーの発言に以下のいずれかが含まれる場合に有効とする。

- `[autodev]`
- `autodev` skill の明示的な利用要求

上記が有効な場合、Agent はその依頼を「autodev skill を起動して、GitHub 上の作業整理、実装、検証、レビュー、merge 判定、報告までを自律的に進める要求」として解釈する。

## 許可される責務

`autodev` が有効な場合、Agent は以下を自律的に実行してよい。

- 既存 Issue / Pull Request / 関連コードの調査
- 必要な Issue の追加・整理
- 実装、テスト追加、検証
- Pull Request の作成と更新
- `docs/dev-workflow/code-review.md` に従った自己レビュー
- merge 可否の判定
- デプロイ後確認とパッチノート形式の報告

## 承認が必要な変更

以下に該当する変更は、ユーザー承認を得るまで実行してはならない。

- インフラコストの増減を伴う変更
- GitHub 上の設定変更
- AWS、Discord Developer Portal など外部サービス上の設定変更
- 認証 / 認可に関わる変更
- DB schema migration または不可逆なデータ変更
- API 互換性を壊す変更
- Secret、CI/CD、権限設定に関わる変更
- 影響範囲が広く、仕様変更を含む可能性が高い変更

## merge 前の必須制約

初回 PR、review 指摘への対応 PR、follow-up PR を区別せず、main に merge する前のすべての PR で以下を必須とする。

- 自己レビュー工程を省略しない
- 重大指摘の有無を確認する
- 残っている merge blocker の有無を確認する
- 上記の結果をユーザーに明示する

ユーザーから「CI確認して続けて」「そのまま進めて」などの指示があっても、自己レビュー完了前に merge してはならない。

## merge 許可条件

以下をすべて満たす場合のみ、Agent は main への merge を行ってよい。

- 関係する検証がすべて成功している
- 対象 PR に必要な check run が成功している
- 自己レビューで重大な指摘が残っていない
- 承認必須条件に該当しない
- 変更が小〜中規模で、影響範囲を説明できる
- デプロイ失敗時に切り戻し可能である

legacy status API の `pending` や空応答だけを根拠に merge 判定してはならない。GitHub 上で確認できる check run を優先する。

上記を満たさない場合、Agent は PR 作成または更新までで停止し、ユーザーに blocker を報告する。

## 報告要件

最終報告はパッチノート形式とし、少なくとも以下を含める。

- 変更内容
- 変更理由
- 影響範囲
- 実施した検証
- 残課題 / 今後の改善候補
